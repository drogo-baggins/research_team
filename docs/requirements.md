# Research Team Agent System — 要件定義

## 概要

ユーザーが設定したテーマに関するディープリサーチ結果を提供するエージェントシステム。

---

## エージェントフレームワーク

- エージェントランタイムとして **pi-agent** を使用する（カスタマイズ性が高く、フレームワークとして広く使われている）
- 品質の高い調査結果を残すために、調査を行うエージェントと、調査タスク分解・タスク管理・成果物品質管理を行うエージェントを設ける
- 期待される品質になるまでタスクを繰り返し、全体の成果物を組み上げていく

---

## Web調査機能

### Human検索モード（デフォルト）

- `web_search`・`web_fetch` において、**実ブラウザの画面ロード確認と取り込み指示のみをユーザーが手動で行う**
- エンジンが不得意とする CAPTCHA や遅延ロード等にも対応できる
- 有償の検索 API サービスが不要

#### 公開情報ポリシー（変更不可）

- 検索エンジンは **Google**（`https://www.google.com/search?q=`）を使用する
- **いかなるサービス・サイトにもログインしない**。認証が必要な情報は取得対象外とする
- これはリサーチ結果が「公開情報に基づく」という信頼性の根拠であり、例外を認めない
- CAPTCHA 等が表示された場合はユーザーが手動で対応し、ログイン操作は行わない

### API検索モード（オプション）

- Serper、Tavily、Brave Search などの検索 API サービスも使用可能

---

## 調査タスク要件

### 調査入力情報

- 調査依頼自体（チャットベース）
- 参照ファイル（依頼と合わせて指定されたテキストや Excel ファイル）

### 作業方法

- 調査ツールは `web_search`、`web_fetch`
- 検索エンジン実体として、API 検索と Human 検索の両方をサポートする
- Python スクリプトをサンドボックス環境で動かし、分析結果の可視化など調査結果を加工するために使用する

### アウトプット

**物理形式**（依頼に従う。依頼がない場合は内容に適合したフォーマットを選択）:
- Markdown テキスト
- PDF
- Excel

**内容の形式**（依頼に従う。依頼がない場合は内容に適合したフォーマットを選択）:
- ビジネス報告
- 学術レポート
- 論文
- 書籍

## エージェントシステムの特徴

### 動的チーム編成

与えられた調査に最適なエージェントを、調査内容に従ってダイナミックにチーム編成する。

初期状態で存在するエージェントは以下の3名のみ:

| エージェント | 役割 |
|---|---|
| **CSM（クライアントサクセスマネージャー）** | ビジネスコンサルタントスキルを持ち、クライアントとのコミュニケーションを一手に引き受ける。期待値管理、解決提案、協力依頼、プロジェクトの状況・成果報告を担う |
| **チームビルダー** | 調査対象に応じた専門家チームの編成・メンバーのスキルセット・ペルソナを定義し、メンバーを育成。プロジェクト運営時もメンバーの働きを確認し、適宜育成やメンバー見直しを行う |
| **PM（プロジェクトマネージャー）** | リサーチ領域でのプロジェクトナレッジを持ち、内部品質基準を定義。品質・スケジュール・マイルストン・手法・リスクを管理し、司令塔の役割を果たす |

### チームワークでの品質向上

- 調査テーマのナレッジに長けた専門家エージェントと、CSM などの非専門家が調査結果に関するディスカッションを行うことで、より顧客視点に近づく調査ができる
- 生の調査成果物をそのまま報告するのではなく、**必ずチーム内のディスカッションを通す**
- 有効なディスカッションが成立することが成果物の品質を高めることを考慮し、チームビルダーはチームの調整を行う
- 情報ソースに裏付けされていない調査は致命的なため、PM によるレビューに加え、ディスカッションの中でも調査の妥当性が議論されるようにする

### プロジェクト管理

- 調査ミッションをプロジェクトとして管理できる
- エージェントチームの管理対象として、保存・リストア・アクティブ切り替えなどが可能
- セキュア設計の観点から、**アーカイブされたものを直接エージェントが編集することは許可しない**

## システムとしての特徴

### ユーザーフレンドリー設計

- まず最初に調査の深さ（規模）を作業開始前にユーザーと認識合わせする
- 調査テーマや規模を調整したのち、調査を担当するチームについてユーザーに報告する
- 調査の全体像である WBS とその進捗が常にユーザーから確認可能。エージェントチームの様々なイベントが通知される
- 調査の方向性が合っていることを示すため、マイルストンで成果物をユーザーに共有する
- いつでもユーザーからのリクエストを CSM が受け付けている
- 調査が誤った方向に進んでしまった場合に備え、チェックポイント毎に一連の成果物を着手前の状態に戻すことができる
- 一度完了した調査に対しても、追加調査や報告体裁の変更などに応じ、物理的・セキュリティ的に許される範囲で柔軟な対応を行う

### セキュリティ設計

- エージェントはプロジェクト直下の `workspace` フォルダのみアクセス可能とする
- 検索語を決めるのは LLM のため、**検索語の汚染チェック**と**取得ページのインジェクション防止**を行う
- 各エージェントの行動、およびエージェント間の交信はログに取られ、予め定義したセキュリティポリシーに違反した行動がないか、専門のエージェントにより事後チェックされる。このログはそのエージェントへの読取権限のみが付与される

### プロジェクト遂行のロバスト設計

- エージェント間での勘違い・それに基づく誤った調査を防ぐため、PM は作業の最初に以下を定義し、エージェントに周知徹底する
  - マイルストン・チェックポイント毎のゴール定義・品質目標、各エージェントのミッション
  - 成果物のファイル名の定義
- 各マイルストンにおいて品質が目標を下回る間は、目標をクリアするまで調査タスクやチーム内のディスカッションを行い、品質課題を解決する
- 目標クリアが難しいと PM が判断した場合、ユーザーとの対話により計画見直しを行う
